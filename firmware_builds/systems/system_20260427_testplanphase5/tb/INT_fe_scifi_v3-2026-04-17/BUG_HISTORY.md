# Bug History

## 2026-04-17

### v3 Qsys cosmetic cleanup still carried dead legacy GUI/IP entries
- Symptom: the v3 Platform Designer systems did not present a clean logical
  view; component order did not match the intended control/data flow, and
  `scifi_datapath_system_v3.qsys` still showed legacy blocks that were no
  longer used by the live v3 source topology.
- Fix: rewrote the v3 `.qsys` GUI ordering metadata so control paths read
  master-to-slave and datapaths read upstream-to-downstream, then removed the
  stale top-level `Use=0` legacy receiver/run-control helper IP from
  `scifi_datapath_system_v3.qsys` while keeping the live RR-mux/FIFO v3
  datapath intact.

### Cosmetic regeneration exposed stale IP packaging paths
- Symptom: after regenerating from the cleaned v3 source systems, Platform
  Designer failed inside nested IP generation even though the source logic was
  otherwise correct.
- Root cause:
  - `ring_buffer_cam_hw.tcl` referenced RTL as `rtl/...` from within its
    `script/` directory, which resolves to the wrong place when cataloged from
    that file.
  - `mutrig_frame_deassembly_hw.tcl` depended on an entry path assumption that
    broke when the catalog reached it through the repo-root symlink.
- Fix:
  - corrected `ring_buffer_cam` RTL fileset paths to `../rtl/...`
  - made `mutrig_frame_deassembly` resolve its icon/version metadata from the
    IP root and use stable root-relative RTL fileset paths
  so `qsys-generate` succeeds from the authoritative repo catalog.

### INT harness reset-seam forces depended on a stale autogenerated bridge name
- Symptom: the v3 RC/SC/datapath integration scripts compiled, but simulation
  aborted before stimulus because the forced datapath-reset seam
  `master_datapath_master_translator_reset_reset_bridge_in_reset_reset` no
  longer existed in the regenerated hierarchy.
- Root cause: fresh Platform Designer generation renamed that internal bridge to
  `mm_pipeline_jtagmaster2rstctrl_reset_reset_bridge_in_reset_reset`.
- Fix: updated the INT scripts to resolve the translator-reset seam dynamically
  from the current hierarchy, accepting either the legacy or regenerated bridge
  name so the harness survives harmless internal renames.

### Missing IP catalog on `feb_system_v3.qsys` open
- Symptom: Platform Designer could not resolve Mu3e IP when opening the FE
  SciFi v3 system from the project.
- Fix: moved to a single authoritative catalog file
  `firmware_builds/mu3e_ip_cores.ipx`, made the project-side `components.ipx`
  wrappers import that catalog, and normalized the catalog paths so the
  systems open without environment-variable-specific absolute paths.

### `scifi_datapath_system_v3` lane source topology regressed from v2
- Symptom: the v3 datapath no longer matched the intended v2 behavior where
  real LVDS traffic and emulator traffic can both feed each MuTRiG datapath.
- Fix: wired each lane as `LVDS + emulator -> RR mux -> shallow FIFO ->
  mutrig_datapath_subsystem_i.decoded_din`, keeping the v2 MM path intact and
  preserving run-control propagation to the emulator instances.

### Upload/link split did not stay aligned with the FE SciFi top-level routing
- Symptom: the v3 upside/downside upload streams were not reliably preserved
  through project integration, which previously caused the FireFly split to be
  wrong.
- Fix: kept the FEB upload contract as:
  - upside link: datapath + slow-control + run-control
  - downside link: datapath only
  and updated the FE SciFi project integration so the two outputs land on
  separate uplink channels / FireFly groups.

### Mixed v2/v3 packaging caused Quartus collisions
- Symptom: the FE SciFi project could see support IP for both FEB generations
  at once, leading to duplicate-IP binding failures.
- Fix: changed project packaging so the active FEB generation is the only one
  compiled into the project build.

### Regression coverage added here
- Added `firmware_builds/tb_int/INT_fe_scifi_v3-2026-04-17/` to preserve a
  repo-local integration harness for v3 run control, datapath E2E, and slow
  control.

### `scifi_datapath_system_v3` run-control management path dropped all RC words
- Symptom: management-injected run-control words reached
  `data_path_subsystem.runctl_mgmt_host_*` but never propagated through the
  datapath splitter, so RC fanout to the v3 datapath was incomplete.
- Root cause: unlike v2, v3 routed `runctl_mgmt_host` through an enabled
  `run_control_mux`. The mux tagged that path with channel `1`, and the
  autogenerated channel adapter on the way into `run_control_splitter` only
  accepted channel `0`, silently dropping the entire stream.
- Fix: restored the v2 behavior by exporting `runctl_mgmt_host` directly into
  `run_control_splitter.in` and disabling the unused mux path in
  `scifi_datapath_system_v3.qsys`.

### Top FEB run-control splitter backpressured reset broadcast
- Symptom: the integrated RC path could leave
  `upload_subsystem_runctl_mgmt_host_valid=1` with host ready low. Earlier
  hardware and simulation probes showed the top splitter ready mask stuck
  with hit-stack leaves deasserted, so a reset/stop-reset broadcast could not
  retire without forcing `run_control_splitter_out[0:15]_ready`.
- Root cause: the top 16-way Altera Avalon-ST splitter was configured with
  `USE_READY=1`, which ANDed all downstream ready leaves into the producer
  ready. That is wrong for this run-control state broadcast: a reset-release
  command must not depend on every reset-affected sink acknowledging in the
  same cycle.
- Fix: changed only the top `run_control_splitter` in
  `scifi_datapath_system_v3_pipe.qsys` to `USE_READY=0` and regenerated the
  pipe systems. Generated per-sink adapters remain for ready-capable slaves,
  while the host sees an unblocked broadcast source. `scripts/run_rc.sh` now
  treats missing `run_control_splitter_outN_ready` nodes as expected and
  requires the unforced host ready plus all 16 valid/data seams to complete.
- Evidence: `run_rc.sh` reports unforced RC fanout acceptance, all 16 datapath
  consumers reached, both SWB boundary links reached, and `Errors: 0`.
  `run_sc.sh` also passes the SC burst-vs-single matrix with all 144 cases
  matched and `Errors: 0` after the same regeneration.

### RC upload seam check in the new INT harness used an exact frame count
- Symptom: the upper upload-path check in `scripts/run_rc.sh` timed out even
  though the injected packet was visible on `upload_data0_sc_rc`.
- Root cause: the SWB ingress stub counts frames on SOP acceptance. By the time
  the harness finished driving a two-word packet and started polling,
  `swb_link*_frames` had already advanced past the exact target value.
- Fix: changed the harness to treat the seam check as monotonic progress
  (`>= expected`) instead of exact equality for both upload links.

### Datapath INT bench used byte addresses on a word-addressed external AVMM port
- Symptom: the v3 datapath smoke bench could observe datapath traffic but
  histogram AVMM accesses never touched the intended histogram windows.
- Root cause: Platform Designer reports the external map in byte addresses, but
  `mm_pipeline_avmm_lvds` is exported as a word-addressed 14-bit AVMM port.
  The bench was using raw `0xA000`/`0xA400` style map values instead of the
  translated bench-side addresses (`0x2800`/`0x2900`).
- Fix: translated the histogram windows to the correct word-addressed bench
  apertures before issuing AVMM reads and writes.

### Datapath INT bench assumed legacy histogram/emulator CSR visibility
- Symptom: the datapath smoke bench could see emulator words, hit-type0 words,
  MTS output, and hit-stack ingress, but histogram configuration never took
  effect and the original external emulator CSR accesses were ineffective.
- Root cause:
  - The embedded `emulator_mutrig_[0..7]` CSRs are only exposed on
    `master_datapath.master` in v3, not on the exported external AVMM bridge.
  - `histogram_statistics_v2` now includes the standard UID/META header at CSR
    words `0..1`, so the legacy alive bench sequence was writing the wrong CSR
    offsets for control, key location, interval, and status.
- Fix:
  - Drove the embedded emulator CSR seam locally inside the datapath smoke
    bench, matching the v2 alive flow rather than pretending those CSRs are
    externally mapped.
  - Shifted the histogram CSR accesses to the current headered map
    (`CONTROL=+2`, `KEY_LOC=+6`, `INTERVAL=+10`, `TOTAL=+13`, etc.) and added
    explicit status readback to the report.

### `scifi_datapath_system_v3` source lost the slow-control aperture to emulator and debug CSRs
- Symptom: the FE SciFi v3 slow-control bridge could reach the main datapath
  CSR windows, but the embedded emulator bank and selected datapath debug/PHY
  management CSRs were still reachable only from the local JTAG master map.
- Root cause: the live source `.qsys` kept those slaves only on
  `master_datapath.master`; the external `mm_pipeline_avmm_lvds.m0` aperture
  did not mirror the emulator CSR bank, `dbg_mm2runctrl_0.csr`, or the
  reset-controller management bridge.
- Fix: mirrored the missing low-address CSRs onto
  `mm_pipeline_avmm_lvds.m0`, keeping the existing external map stable while
  making the emulator and the remaining datapath-side AVMM slaves reachable
  through the SC hub path.

### `mutrig_datapath_system_v3` advertised a dead LVDS-error export
- Symptom: trying to expose per-lane LVDS error counters through the v3
  datapath parent produced dangling-interface generation failures.
- Root cause: `mutrig_datapath_system_v3.qsys` still exported
  `avst_in`/`lvdserr_counter`, but the backing `lvdserr_count_subsystem` was
  disabled and not part of the live source topology.
- Fix: removed the dead exported interfaces instead of pretending the block was
  usable. The SC-facing bridge now covers the full live datapath AVMM set, and
  LVDS-error counting remains a separate follow-up if that subsystem is ever
  intentionally re-enabled.

## 2026-04-25

### MTS overflow lookback was coupled to the main timestamp buffer latency
- Symptom: phase-4 emulator tests could produce MTS timestamp errors when the
  expected pipeline latency was increased to cover the live datapath buffering.
- Root cause: the same generic controlled both the timestamp buffer expected
  latency and the overflow-disambiguation lookback window, so a latency tuning
  change widened the overflow check beyond the valid comparison window.
- Fix: split `MUTRIG_OVERFLOW_LOOKBACK_8N` from
  `MUTRIG_BUFFER_EXPECTED_LATENCY_8N`, kept the live latency at 4096 words,
  and set the overflow lookback to 2000 words in the pipe Qsys image.
- Evidence: standalone MTS tests and the system datapath smoke test pass with
  `+TB_DP_REQUIRE_NO_MTS_ERRORS`.

### MTS debug counters counted held valid data while ready was low
- Symptom: after a low-rate phase-4 run ended, MTS1 `discard_hits` and
  `total_hits` continued to increase even though the run was stopped; SC and
  run-control were otherwise healthy.
- Root cause: `mts_processor` incremented total/discard debug counters on
  `asi_hit_type0_valid` instead of the accepted ready/valid transfer. When the
  upstream mux held a buffered valid beat while `asi_hit_type0_ready=0`, the
  debug counters re-counted that beat.
- Fix: registered the internal ready signal and gated total/discard counter
  updates with `asi_hit_type0_valid and asi_hit_type0_ready_i`. The IP version
  is now `26.0.7`.
- Evidence: the directed standalone MTS test holds invalid data with ready low
  in IDLE and verifies no counter movement; system DP smoke passes; hardware
  phase-4 at `0x0100` passes 2/2 with `MTS Discard = 0`, and a post-end direct
  MTS1 read is stable.

### Histogram post-stack tap FIFO saturated at the Phase 4 default rate
- Symptom: SC Phase 1/2 and RC Phase 3 passed, MTS discard stayed at zero, and
  hit-stack frame counters matched, but Phase 4 at `hit_rate=0x0800` reported
  `DROPPED_HITS` deltas of roughly 450 per 100 ms.
- Root cause: the live pipe Qsys image uses one passive post-stack histogram tap
  (`histogram_ingress_bridge_0.hist_out -> histogram_statistics_0.hist_fill_in`)
  with `N_PORTS=1`; the seven `hist_rate_splitter_N` inputs mentioned in the
  test plan are not present. Bursty post-stack traffic filled the 256-entry
  ingress FIFO. The CSR also truncated a 9-bit full FIFO level into the 8-bit
  `PORT_STATUS.fifo_level_max` field, so a full level of 256 appeared as zero.
- Fix: parameterized the histogram ingress FIFO depth, set the pipe instance to
  `FIFO_ADDR_WIDTH=10` (1024 entries), and changed `PORT_STATUS.fifo_level_max`
  to saturate at `0xFF` instead of wrapping.
- Evidence: standalone histogram test `P03_wire_burst_absorb` drives a 1024-hit
  one-hit-per-clock burst into port 0 and requires `TOTAL_HITS=1024`,
  `DROPPED_HITS=0`, and saturated `PORT_STATUS.fifo_level_max`.
