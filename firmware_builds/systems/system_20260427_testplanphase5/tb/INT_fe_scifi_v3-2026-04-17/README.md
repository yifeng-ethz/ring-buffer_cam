# FE SciFi V3 Integration Harness

This integration harness lives with the Qsys sources and exercises the current
`feb_system_v3` source-level integration.

Scope:
- Run-control fanout and dual upload-link delivery through `feb_system_v3`
- Slow-control `sc_hub` reachability across control and datapath slaves, with
  both single-word and burst access
- Datapath end-to-end hit generation through the embedded `emulator_mutrig_[0..7]`
  instances in `scifi_datapath_system_v3`, including histogram uptake on the
  v3 datapath histogram path

Entry points:
- `scripts/run_rc.sh`
- `scripts/run_sc.sh`
- `scripts/run_dp_e2e.sh`
- `scripts/run_all.sh`

Layout:
- `tb_src/`: stable repo-local stubs and datapath-native benches
- `generated/`: derived v3 bench copies and quiet adapter/top wrappers
- `cases/`: per-domain stimulus inventory and future expansion buckets
- `docs/`: harness architecture and handoff notes for larger DV/UVM work
- `REPORT/`: runtime logs grouped by `rc/`, `sc/`, and `dp/`
- `work/`: simulator libraries and transient build products

Dependencies:
- QuestaOne 2026 at `${QUESTA_HOME}`. The supported host setup is the repo
  wrapper `../../../scripts/questa_one_env.sh`, which resolves to
  `/data1/questaone_sim/questasim` on this machine.
- The FE SciFi firmware tree at
  `/home/yifeng/packages/online_dpv2/online/fe_board/fe_scifi`
  for shared packages and the authoritative v2-era FEB integration benches

Notes:
- The RC and SC cases derive temporary v3 bench copies from the proven
  `online_dpv2` FEB benches, then compile them against
  `firmware_builds/feb_system_v3/synthesis/feb_system_v3.qip`.
- The datapath case is a repo-local v3-native smoke bench.
- The current harness is intentionally organized as a bridge toward future UVM
  work: domain case inventories live under `cases/`, while generated bench
  transforms are isolated under `generated/` instead of the simulator worktree.
- The scripts resolve the internal datapath translator-reset seam dynamically,
  because Platform Designer may rename that autogenerated bridge across clean
  regenerations without changing behavior.
- The external datapath AVMM bridge is word-addressed even though Platform
  Designer reports byte-style addresses in the generated map; the bench uses
  translated word addresses for histogram access.
- The embedded emulator CSRs are not exported on that external AVMM bridge in
  v3, so the smoke bench drives the local emulator CSR seam directly and uses
  the external AVMM path for histogram access and datapath-side MM checks.
- Simulation reports are written under `REPORT/{rc,sc,dp}/`.
